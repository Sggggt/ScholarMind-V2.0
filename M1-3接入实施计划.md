# ScholarMind Mobile 接入 M1-3 实施计划

## 1. 目标

本计划的目标不是单独再造一套移动端任务系统，而是让 `mobile/` 真正接入当前 ScholarMind 的真实后端，使手机端可以完成以下闭环：

1. 在手机端创建研究任务。
2. 在手机端实时查看 `M1 文献调研 → M2 研究空白 → M3 Idea 生成与打分` 的进度。
3. 在 `M3` 完成后，于手机端查看候选 Idea、继续生成更多 Idea、选择一个 Idea 推进到 `M4`。
4. 让桌面端与手机端通过同一套后端任务状态、产物和日志保持同步，而不是手机和电脑直接点对点通信。

这意味着“手机端与桌面端通信”的正确实现方式是：

- 两端共享同一个 FastAPI 后端作为唯一事实源。
- 两端通过相同的 REST API 和 WebSocket 订阅同一个 `task_id`。
- 任务状态变化由后端广播，桌面端和手机端各自更新本地状态。

## 2. 现状审计

### 2.1 后端真实情况

当前后端已经具备移动端接入 `M1-3` 的核心能力：

- `POST /api/tasks` 创建任务并自动从 `M1` 开始执行。
- `GET /api/tasks`、`GET /api/tasks/{id}`、`GET /api/tasks/{id}/logs` 可获取任务与日志。
- `GET /api/tasks/{id}/ideas`、`POST /api/tasks/{id}/continue-ideas`、`POST /api/tasks/{id}/select-idea` 支持 `M3` 结束后的 Idea 管理。
- `GET /api/tasks/{id}/artifact-content` 可读取 `m1_literature_review.md`、`m2_gap_analysis.json`、`m3_scored_ideas.json` 等产物。
- `GET /api/tasks/{id}/output`、`GET /api/tasks/{id}/artifacts` 为后续 `M4+` 扩展留好了入口。
- WebSocket 真实地址是 `ws://{host}:8000/ws/{task_id}`，不是模板风格的 `/api/trpc` 或 `/api/ws/...` 路径。

更关键的是，当前后端的真实流程与原始分工文档不同：

- 任务创建后会自动执行 `M1 → M2 → M3`。
- `M3` 完成后，任务会被置为 `paused`，等待用户选择某个 Idea。
- 只有调用 `POST /api/tasks/{id}/select-idea` 后，任务才会从 `M4` 继续执行。

所以手机端第一阶段不应该尝试“一键跑完整个 M1-M9”，而应该先把 `创建任务 + 观察 M1-M3 + 选择 Idea 推进` 做扎实。

### 2.2 桌面端真实情况

桌面端 `react-client/` 已经在对接这套 FastAPI：

- REST 层使用 `/tasks`、`/tasks/{id}`、`/tasks/{id}/select-idea` 等真实接口。
- WebSocket 使用 `/ws/{task_id}`。
- `IdeaGenerationPage` 已经围绕 `M3 完成后暂停、选择 Idea 再推进` 这一逻辑实现。
- `WorkspaceSync` 说明仅靠 WebSocket 还不够，桌面端还用了“收到消息后的短延迟刷新 + 断线轮询”的策略来补足状态同步。

这给移动端的实现提供了明确参考：移动端应复用这套同步思路，而不是重新发明一套通信协议。

### 2.3 mobile 当前情况

`mobile/` 目前更像一个完成了 UI 壳子的独立模板应用，而不是 ScholarMind 的真实移动客户端：

- 页面结构基本齐全，列表、创建、详情、日志、评审、设置页都已经搭好。
- 状态管理、演示数据、任务详情展示都已初步实现。
- 但 `lib/api.ts` 仍然请求 `/api/trpc/...`，接的是 `mobile/server/` 这套模板内置 tRPC 服务。
- `mobile/server/` 的数据模型、任务执行逻辑、数据库结构，与 ScholarMind 当前后端不是同一套系统。
- 当前 `mobile` 中的“部分流程选择”“评审结果”“模块 I/O”很多还是演示态或模板态，并未接入真实任务产物。

结论：移动端最大工作量不是补页面，而是把通信底座从模板 tRPC 切换到 ScholarMind FastAPI，并将页面能力收敛到当前后端真实支持的流程。

## 3. 范围定义

### 3.1 第一阶段纳入范围

第一阶段只纳入 `M1-3` 必需能力：

1. 手机端连接 ScholarMind 后端。
2. 手机端创建任务。
3. 手机端查看任务列表与任务详情。
4. 手机端实时接收 `M1-3` 执行进度。
5. 手机端查看 `M1` 文献综述、`M2` 缺口分析、`M3` 候选 Idea。
6. 手机端在 `M3` 完成后执行：
   - 查看 Idea 列表
   - 继续生成更多 Idea
   - 选择 Idea 推进到 `M4`
7. 手机端和桌面端对同一任务保持同步。

### 3.2 第一阶段暂不纳入

以下内容建议明确延后，避免把项目拖入大而全：

- 手机端本地自带 `mobile/server/` 继续作为任务后端。
- 手机端直接操作本地代码仓库或本地文件系统。
- 手机上完整承载 `M4-M9` 的重型代码/论文编辑体验。
- 在移动端保留“任意模块自由组合执行”的假能力。

原因很简单：当前真实后端并不支持移动端创建任务时自由指定“从哪些模块开始跑”，而 UI 里这部分配置会制造假功能和错误预期。

## 4. 目标架构

### 4.1 总体架构

```text
Mobile App  ─┐
             ├── REST API + WebSocket ──> FastAPI Backend ──> Workspace / DB / Pipeline
Desktop App ─┘
```

原则：

- 后端是唯一事实源。
- 手机端和桌面端不直接互相调用。
- 两端通过共享 `task_id` 观察和控制同一个任务。

### 4.1.1 网络接入模式

手机端连接电脑上的后端，建议支持两种网络模式：

1. 局域网直连
   - 适用于手机和电脑在同一 Wi-Fi/同一网段。
   - 手机端直接配置电脑内网地址，例如 `http://192.168.1.100:8000`。
2. 内网穿透
   - 适用于手机和电脑不在同一局域网。
   - 电脑端运行穿透客户端，把本地 `8000` 端口映射到公网地址。
   - 手机端配置穿透后的公网地址，例如 `https://scholarmind.example.com`。

推荐策略：

- 开发联调优先使用局域网直连。
- 异地演示、外网访问、移动网络访问时使用内网穿透。

### 4.2 同步模型

建议采用“WebSocket 为主，轮询兜底”的双通道方案：

1. 任务详情页打开时：
   - 先 `GET /api/tasks/{id}` 拉一次快照。
   - 再连接 `ws://{host}:8000/ws/{id}`。
2. 收到 WebSocket 消息后：
   - 先更新本地进度和日志。
   - 延迟 1-2 秒再调用一次 `GET /api/tasks/{id}`，确保拿到真实状态。
3. WebSocket 断线或应用从后台恢复时：
   - 对运行中任务启动 5 秒轮询。

这样做的原因是：

- 当前 WebSocket 主要广播进度和产物，不是完整状态快照。
- `M3` 结束后任务会变为 `paused`，单靠当前消息流不一定能完整反映状态变化。
- 桌面端已经采用类似策略，移动端直接复用风险最低。

### 4.3 手机端与桌面端“通信”的定义

这里的“通信”分为三类：

1. 状态同步
   - 手机创建任务，桌面端任务列表能看到新任务。
   - 桌面选择 Idea，手机端任务详情能看到状态从 `paused@M3` 变为 `running@M4`。
2. 产物同步
   - `m1_literature_review.md`、`m2_gap_analysis.json`、`m3_scored_ideas.json` 由后端统一存储，两端都读同一份。
3. 控制同步
   - 任一端执行 `pause/resume/abort/select-idea`，另一端通过 WS + 刷新感知到变化。

结论：不需要做手机和桌面的专门“通信层”，只需要让两端严格遵循后端契约。

### 4.4 内网穿透技术实现

如果后端运行在个人电脑上，而手机端需要在外网访问，则可在电脑侧部署内网穿透。

实现原则：

- 暴露的是电脑上的 `FastAPI 8000` 端口。
- 手机端只感知“后端基地址”，不感知内网穿透实现细节。
- WebSocket 必须与 REST 一起可用。

推荐实现方案分级如下：

1. 开发测试：`ngrok`
   - 优点：配置最快，几分钟可跑通。
   - 缺点：地址不稳定，长期使用成本高。
2. 稳定演示：`cloudflared tunnel`
   - 优点：支持稳定域名、HTTPS、配置相对简单。
   - 缺点：需要域名或 Cloudflare 环境。
3. 长期自托管：`frp`
   - 优点：可控性高，适合团队长期使用。
   - 缺点：需要自有公网服务器。
4. 私有组网：`Tailscale`
   - 优点：安全性高，适合团队成员设备互通。
   - 缺点：更偏组网，不是传统公网暴露。

第一阶段建议：

- 团队本地开发：局域网直连。
- 需要校外/异地演示：先用 `ngrok` 或 `cloudflared tunnel`。

实现要求：

1. 穿透地址必须同时支持：
   - `https://.../api/tasks`
   - `wss://.../ws/{task_id}`
2. 手机端配置项统一存一个 `backend_base_url`。
3. WebSocket 地址由 `backend_base_url` 自动推导：
   - `http://` -> `ws://`
   - `https://` -> `wss://`
4. 连接测试必须覆盖：
   - REST 可达
   - WebSocket 可建立连接

推荐的落地方式：

- 在后端电脑上新增一个启动脚本，例如：
  - `start_backend.bat`
  - `start_tunnel.bat`
- 或在文档里明确两步启动：
  1. 启动 FastAPI
  2. 启动隧道

示例一：`ngrok`

```bash
ngrok http 8000
```

示例二：`cloudflared`

```bash
cloudflared tunnel --url http://localhost:8000
```

示例三：`frpc`

```ini
[scholarmind]
type = tcp
local_ip = 127.0.0.1
local_port = 8000
remote_port = 8000
```

安全要求：

- 不允许把后端裸暴露到公网长期使用且无鉴权。
- 如果使用内网穿透，至少要增加：
  - 后端访问令牌或 Bearer Token
  - 简单白名单或分享口令
  - 基础限流
- LLM API Key 仅保留在电脑后端，不下发到手机端。

## 5. API 契约整理

移动端第一阶段应只接入以下真实接口。

### 5.1 任务与状态

- `POST /api/tasks`
- `GET /api/tasks`
- `GET /api/tasks/{id}`
- `POST /api/tasks/{id}/pause`
- `POST /api/tasks/{id}/resume`
- `POST /api/tasks/{id}/abort`
- `GET /api/tasks/{id}/logs?limit=200`

### 5.2 M1-M3 产物

- `GET /api/tasks/{id}/artifact-content?path=m1_literature_review.md`
- `GET /api/tasks/{id}/artifact-content?path=m1_sources.json`
- `GET /api/tasks/{id}/artifact-content?path=m2_gap_analysis.json`
- `GET /api/tasks/{id}/artifact-content?path=m3_scored_ideas.json`
- 或直接使用 `GET /api/tasks/{id}/ideas`

建议：

- `M1`、`M2` 通过 `artifact-content` 读取内容。
- `M3` 优先使用 `GET /api/tasks/{id}/ideas`，因为这是明确面向 Idea 阶段的接口。

### 5.3 M3 交互

- `GET /api/tasks/{id}/ideas`
- `POST /api/tasks/{id}/continue-ideas`
- `POST /api/tasks/{id}/select-idea`

### 5.4 实时通信

- `GET /ws/{task_id}` 不是 HTTP 接口。
- WebSocket 真实地址为 `ws://{host}:8000/ws/{task_id}`。

当前需要处理的消息类型：

- `progress`
- `result`
- `error`
- `need_review`
- `completed`

移动端要特别关注：

- `progress` 中 `module=M3` 且阶段结束后，要主动刷新任务状态和 Idea 列表。
- `completed/error` 到来时，要刷新详情并触发本地通知。

## 6. 移动端代码改造计划

### 6.1 阶段 A：替换通信底座

目标：让 `mobile` 从模板 tRPC 切到 ScholarMind FastAPI。

需要改的核心文件：

- `mobile/lib/api.ts`
- `mobile/lib/types.ts`
- `mobile/lib/task-store.ts`
- `mobile/lib/task-provider.tsx`
- 新增 `mobile/lib/websocket.ts`

具体动作：

1. 删除或停用 `lib/api.ts` 里所有 `/api/trpc/...` 调用。
2. 改为直接请求 FastAPI 的 `/api/tasks` 等接口。
3. 统一 `Task`、`ModuleState`、`LogEntry`、`ReviewResult` 类型字段，和 `backend/api/schemas.py` 对齐。
4. 在 `TaskProvider` 中加入：
   - `fetchTaskIdeas`
   - `continueIdeas`
   - `selectIdea`
   - `fetchArtifactContent`
5. 新增 WebSocket 客户端：
   - 支持连接任务级 `ws/{task_id}`
   - 支持自动重连
   - 支持前后台切换后的重连
   - 支持收到消息后 dispatch 到 store

验收标准：

- 设置页配置后端地址后，任务列表能拉到真实任务。
- 创建任务后，详情页能看到真实任务状态而不是 demo 数据。

### 6.2 阶段 B：收敛创建页，只保留真实支持能力

目标：让创建页与后端真实能力一致。

当前问题：

- `create.tsx` 里有“完整流程 / 部分流程 / 自选模块”能力。
- 但真实后端 `POST /api/tasks` 当前只支持创建任务并从 `M1` 开始跑。
- 如果继续保留部分流程选择，用户会误以为手机端能自由指定从 `M4` 或 `M8` 开始。

建议改法：

1. 第一阶段把“部分流程”入口隐藏。
2. 创建页只保留：
   - 研究主题
   - 补充描述
   - 可选配置项，如 `max_ideas`
3. 如果要保留“高级选项”，只展示当前后端真实可消费的配置项。

验收标准：

- 用户创建的任务一定能被后端正确接受并自动进入 `M1`。
- 页面上不再出现与后端能力不一致的控制项。

### 6.3 阶段 C：任务详情页接入真实 M1-M3

目标：让详情页成为 `M1-3` 的主工作台。

建议改造方向：

1. 顶部状态卡显示：
   - 任务状态
   - 当前模块
   - 当前 step
   - 更新时间
2. 模块时间线显示：
   - `M1` 文献调研
   - `M2` 研究空白
   - `M3` Idea 打分
3. 模块展开后不再显示 demo I/O，而是显示真实产物摘要：
   - `M1`：文献综述摘要 + 文献数量
   - `M2`：研究空白列表
   - `M3`：Idea 数量、最佳 Idea、是否已暂停等待选择
4. 保留 `pause/resume/abort`。
5. 当任务到达 `paused + current_module=M3` 时，详情页显示“进入 Idea 选择”按钮。

建议新增数据加载逻辑：

- 详情页加载时并行获取：
  - `GET /api/tasks/{id}`
  - `GET /api/tasks/{id}/logs`
  - 当前模块对应的 artifact 或 ideas

验收标准：

- 任务从创建到 `M3 paused` 的全过程，可以只在任务详情页看明白。

### 6.4 阶段 D：新增 M3 Idea 选择页

目标：把 `M3` 做成真正可操作的移动端节点。

建议新增页面：

- `mobile/app/task/[id]/ideas.tsx`

页面能力：

1. 拉取 `GET /api/tasks/{id}/ideas`
2. 展示每个 Idea 的核心字段：
   - 标题
   - premise / hypothesis
   - innovation
   - feasibility
   - evidence strength
   - risk
   - composite score
3. 提供两个动作：
   - “继续生成更多 Idea” -> `POST /api/tasks/{id}/continue-ideas`
   - “选择该 Idea 并推进” -> `POST /api/tasks/{id}/select-idea`
4. 选择成功后：
   - 刷新任务详情
   - 返回任务详情页
   - 触发 toast：已推进到 M4

这是第一阶段最关键的页面，因为它是 `M1-3` 与后续流程的桥。

验收标准：

- 用户可以在手机上完成 `M3` 结束后的决策，不需要回到桌面端。

### 6.5 阶段 E：日志与通知

目标：让手机端成为“轻控制 + 轻监控”终端。

日志页改造：

1. 初次进入时拉 `GET /api/tasks/{id}/logs`
2. WebSocket 消息实时追加到日志列表
3. 保留 `INFO/WARN/ERROR` 过滤

通知改造：

建议新增 `mobile/lib/notifications.ts`，用 `expo-notifications` 处理：

- `M3` 完成并暂停等待选择时通知
- `need_review` 到来时通知
- `completed` 到来时通知
- `error` 到来时通知

注意：

- iOS/Android 的后台行为有限，不能假设 App 长时间后台驻留仍能保持 WebSocket。
- 所以本地通知主要在 App 前台收到事件时触发，后台推送属于下一阶段。

### 6.6 阶段 F：设置页与连接诊断

目标：让用户能正确把手机连到桌面/后端所在机器。

设置页需要修正：

1. `testConnection` 不应再请求 `/api/trpc/tasks.list`。
2. 改为请求 `GET /api/tasks` 或新增一个轻量 `/health`。
3. 清楚提示：
   - 手机和电脑需在同一局域网
   - 手机填的是运行 FastAPI 的主机地址
   - 例如 `http://192.168.1.100:8000`
   - 如果使用内网穿透，则填写公网地址，例如 `https://xxx.ngrok-free.app`

建议后端补一个轻量健康检查接口：

- `GET /api/health -> { ok: true }`

这样设置页测试连接会更干净。

设置页还应补充一个“连接模式说明”区块：

1. 局域网模式
   - 用电脑内网 IP 连接。
2. 内网穿透模式
   - 用隧道提供的公网地址连接。
   - 如果地址是 `https://`，App 必须自动改用 `wss://` 建立 WebSocket。

建议设置页连接诊断顺序：

1. 校验 URL 格式。
2. 请求 `GET /api/health` 或 `GET /api/tasks`。
3. 进行一次 WebSocket 握手测试。
4. 显示连接类型：
   - `局域网`
   - `公网/穿透`

建议新增持久化字段：

- `backend_base_url`
- `resolved_ws_url`
- `network_mode`

## 7. 桌面端与手机端联动方案

### 7.1 基础原则

桌面端和手机端都订阅同一个任务，而不是彼此订阅。

联动路径如下：

1. 手机创建任务
   - 后端落库并启动 pipeline
   - 桌面端刷新列表或订阅全局任务变化后可见
2. 后端推进到 `M1/M2/M3`
   - 两端各自通过 `ws/{task_id}` 收到进度
3. 手机选择 Idea
   - 调用 `select-idea`
   - 后端从 `M4` 重新启动
   - 桌面端刷新后进入 `repository/code generation`
4. 桌面端暂停/终止任务
   - 手机详情页同步看到状态变化

### 7.2 推荐的同步责任划分

- 后端负责：
  - 持久化状态
  - 推送执行事件
  - 保存产物
- 桌面端负责：
  - 完整工作流可视化
  - 深度查看代码/论文/实验
- 手机端负责：
  - 快速创建
  - 进度跟踪
  - M3 决策
  - 轻量日志与告警

这样分工最符合当前生态，不会把移动端做成桌面端缩小版。

## 8. 需要后端配合的改动

移动端可以先按现有接口接入，但为了稳定联调，建议后端补以下优化。

### 8.1 补健康检查

新增：

- `GET /api/health`

用途：

- 设置页测试连接
- 启动诊断
- 自动重连自检

### 8.2 WebSocket 补状态字段

建议在每条 WS 消息 `data` 中增加：

- `task_status`
- `current_module`

原因：

- 当前移动端仅靠 `progress` 消息很难准确知道任务是否已进入 `paused/review`。
- 如果消息直接带状态，移动端就能少一次刷新依赖。

### 8.3 在状态变化时补事件

建议在以下状态变化时主动推一条状态型事件，或至少触发一次 `progress/result` 风格广播：

- `running -> paused`，尤其是 `M3` 完成后暂停
- `paused -> running`
- `running -> review`
- `running -> failed`

如果不想新增消息类型，也可以统一使用 `progress`，但需要保证前端能感知状态切换。

### 8.4 契约统一

后端 schema 一旦变更，需要同步更新：

- `react-client/src/types/backend.ts`
- `mobile/lib/types.ts`

建议把这两处的任务/日志/WS 类型抽成共享契约，避免桌面端和移动端再次分叉。

### 8.5 内网穿透部署配合

为了让“电脑跑后端、手机远程访问”真正可用，建议后端侧补一份运维说明或脚本。

建议补充内容：

1. 本地启动脚本
   - 启动 FastAPI：监听 `0.0.0.0:8000`
2. 穿透启动脚本
   - `ngrok http 8000`
   - 或 `cloudflared tunnel --url http://localhost:8000`
3. 健康检查脚本
   - 本地自检 `http://127.0.0.1:8000/api/health`
   - 外网自检穿透后的地址
4. CORS 与 WebSocket 校验
   - 明确穿透后 `https/wss` 场景可正常工作

如果后续要长期对外提供移动访问，建议增加：

- `Authorization: Bearer <token>` 支持
- 简单用户体系或设备绑定
- 请求速率限制
- 访问日志

## 11. 风险与规避

### 风险 1：mobile 继续沿用模板 tRPC

后果：

- 手机端会维护第二套任务系统。
- 无法与桌面端共享同一个任务状态。

规避：

- 第一阶段明确停用 `mobile/server/` 的任务业务。

### 风险 2：UI 继续暴露后端不支持的“部分流程”

后果：

- 用户创建出后端无法正确执行的任务。
- 前端体验与真实能力错位。

规避：

- 创建页只保留真实支持的字段。

### 风险 3：只靠 WebSocket 不做刷新

后果：

- `M3 paused` 这类状态变化可能不同步。

规避：

- 采用“WS + 延迟刷新 + 断线轮询”三段式同步。

### 风险 4：把手机端做成桌面端完整替代

后果：

- 工期膨胀。
- 交互复杂度超出手机使用场景。

规避：

- 第一阶段只做 `轻控制 + 轻查看 + M3 决策`。

### 风险 5：内网穿透只打通 HTTP，未打通 WebSocket

后果：

- 手机端能打开列表，但无法实时同步任务进度。

规避：

- 设置页连接测试必须包含 WebSocket 握手测试。
- 文档中明确 `https -> wss` 的地址推导规则。

### 风险 6：公网暴露无鉴权

后果：

- 后端接口可能被未授权访问。
- 任务、日志和模型调用存在泄露风险。

规避：

- 通过访问令牌、限流和白名单保护穿透入口。
- 不在手机端存储任何模型密钥。

## 12. 推荐落地顺序

最推荐的实际执行顺序如下：

1. 先改 `mobile/lib/api.ts` 和 `mobile/lib/types.ts`。
2. 再接 `TaskProvider` 的真实任务读写。
3. 接入任务详情页的真实数据。
4. 新增 `ideas.tsx` 完成 `M3` 决策闭环。
5. 最后补 WebSocket、通知和联调细节。

这样做的原因是：

- 没有真实 API 层，后面的页面接入都会是假工作。
- `M3` 是 `M1-3` 阶段真正的决策出口，Idea 页必须尽早落地。
- 通知、动效、体验优化都应该放在闭环打通之后。

## 13. 最终交付定义

当以下条件全部满足时，可以认为“手机端已经接入 M1-3，并与桌面端通信”：

1. 手机端不再依赖 `mobile/server/` 作为任务主后端。
2. 手机端可以连接 ScholarMind FastAPI。
3. 手机端可以创建真实任务并看到 `M1-3` 进度。
4. 手机端可以读取 `M1/M2/M3` 的真实产物。
5. 手机端可以在 `M3` 选择 Idea 并推进到 `M4`。
6. 桌面端与手机端对同一任务的状态、日志和产物保持同步。
7. 断线、空态、错误态都有可接受的处理。
