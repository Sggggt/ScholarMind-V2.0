# ScholarMind 产品亮点与竞品分析

> 面向学术研究全流程的 AI 原生科研助手——从选题到论文评审的一站式自动化平台。

---

## 1. 产品定位

**一句话定位：** ScholarMind 让研究者用一句自然语言描述研究意图，即可自动完成「文献调研 → 空白识别 → 构思生成 → 代码实现 → 实验执行 → 论文撰写 → 模拟评审」的完整科研闭环。

**核心主张：** 不是给研究者一个聊天框，而是给他们一支 **AI 科研团队**——每个模块对应科研流程中的一个角色（调研员、方法论专家、工程师、实验员、审稿人），模块间自动传递上下文，用户只需在关键节点做决策。

## 2. 目标用户与痛点

### 2.1 核心用户画像

| 用户群体 | 典型场景 | 核心痛点 |

### 3.2 核心差异化优势

#### 优势一：全流程闭环 vs 碎片化工具拼接

```
传统科研工作流 (碎片化):
  用户 ──► Google Scholar (查文献)
       ──► ChatGPT (讨论 idea)
       ──► GitHub Copilot (写代码)
       ──► 本地终端 (跑实验)
       ──► Overleaf (写论文)
       ──► 同事/导师 (审稿)
  问题: 上下文断裂，需反复复制粘贴，无自动衔接

ScholarMind 工作流 (全流程闭环):
  用户 ──► 一句话描述 ──► M1→M2→M3→M4→M5→M6→M7→M8→M9
         (上下文在模块间自动传递，每个环节都基于前序产出)
  优势: 零切换成本，端到端可追溯
```

#### 优势二：AI Agent 自主执行 vs 单轮问答

```
传统 AI 助手:
  用户 → LLM → 文本回复 → 用户手动执行 → 粘贴结果 → LLM → ...
  (用户是执行者，AI 只是顾问)

ScholarMind:
  用户 → 描述意图 → Agent 自主规划 → Agent 自主编码(Aider)
       → Agent 自主执行实验 → Agent 自主分析 → Agent 自主迭代
       → 用户仅在关键节点介入 (选择 idea / 审阅结果)
  (AI 是执行者，用户是决策者)
```

#### 优势三：增量式交互 vs 一次性生成

```
传统工具:
  提交请求 → 等待 → 得到最终结果 (中间过程不可见、不可控)

ScholarMind:
  M3 阶段: 每生成一个 idea 就实时推送，用户可随时选择推进
  M7 阶段: 实验未达标自动重试，用户全程可见进度
  全程: WebSocket 实时推送每个步骤的进度和日志
  (用户始终掌握主动权，而非等待黑箱)
```

#### 优势四：移动端原生体验

```
竞品现状:
  - ChatGPT / Claude: 移动端是 Web 包装
  - Semantic Scholar / Elicit: 仅 Web
  - AI-Scientist: 仅 CLI

ScholarMind:
  - Expo 原生移动端: 实时推送、手势交互、离线缓存
  - 桌面端: Next.js 完整管理界面
  - mDNS 局域网自动发现: 手机自动连接电脑上的后端
  - 场景: 用户在通勤时用手机查看实验进度、选择 idea、阅读论文
```

### 4.2 学术诚信保障

| 机制 | 说明 |
|------|------|
| **文献引用可追溯** | M1 保留所有来源 URL 和 paper 元数据；M8 论文写作插入真实 BibTeX 引用 |
| **实验结果可复现** | 所有代码通过 git 版本管理；实验使用固定随机种子 (seed)；结果文件完整保留 |
| **多级降级策略** | 每个模块主路径失败后降级到更保守的方案，而非编造数据 |
| **静态代码门禁** | experiment_guard 检查生成代码是否包含 idea 关键词和要求的指标，防止文不对题 |
| **LLM 数据标注** | M6 的 LLM 模拟数据明确标记为 `type: "llm_simulated"`，与真实实验结果区分 |
| **评审包含可信度评估** | M9 的 credibility_check 量化评估整个流程的完整度和可追溯性 |
| **全流程日志** | TraceLog 记录每个模块的每一步操作，支持审计回溯 |

### 5.2 关键鲁棒性指标

| 场景 | 处理方式 | 用户感知 |
|------|---------|---------|
| LLM API 超时 | 自动重试 + 指数退避 | 无感知（除非连续6次失败） |
| LLM API 限流 (429) | 解析 Retry-After + 等待后重试 | 可能稍慢，但不中断 |
| GPT-Researcher 不可用 | 降级到本地 Semantic Scholar + LLM | 综述质量略降，流程不中断 |
| PaperQA2 不可用 | 降级到纯 Semantic Scholar 搜索 | 分析精度略降，流程不中断 |
| Aider 不可用 | LLM 全量重写 experiment.py | 代码质量略降，功能完整 |
| SSH 服务器不可用 | 自动降级到本地执行 + LLM 模拟 | 实验精度取决于本地环境 |
| 实验代码运行失败 | Aider 自动修改代码后重试 | 最多重试 MAX_ITERS 次 |
| 实验结果未达标 | 自动回退 M6 重新实验 | 最多重试 max_retries 次 |
| 后端进程重启 | 自动恢复 running/paused/review 状态任务 | 任务从断点继续，不丢失进度 |
| 用户中途暂停 | 立即中断当前操作，保存断点 | 可随时恢复继续 |

## 7. 技术亮点总结

### 7.1 六大技术创新点

```
┌──────────────────────────────────────────────────────────────────────┐
│                                                                      │
│  ① 增量式 Idea 生成                                                  │
│     M3 每生成一个 idea 就实时推送并保存快照                            │
│     用户无需等待全部生成完毕，随时选择推进                              │
│     支持 "继续生成更多" 的增量模式                                    │
│                                                                      │
│  ② 两级 RAG (检索增强生成)                                           │
│     粗粒度 Web RAG (GPT-Researcher) + 细粒度文档 RAG (PaperQA2)      │
│     比纯 LLM 生成更 grounded，比纯检索更综合                          │
│                                                                      │
│  ③ 自主编码 Agent (Aider)                                            │
│     不是让 LLM 生成代码片段让用户复制                                  │
│     而是 Aider Agent 自主修改代码文件，支持 diff/whole 两种编辑模式   │
│     失败时自动从 diff 降级到 whole                                    │
│                                                                      │
│  ④ 渐进降级策略                                                      │
│     每个模块都有 3 层降级路径                                         │
│     确保在任何环境下（无GPU、无API Key、网络受限）都能完成基本流程     │
│                                                                      │
│  ⑤ 任务级模型热切换                                                  │
│     runtime_config 通过 ContextVar 实现线程安全的任务级配置           │
│     不同任务可同时使用不同 LLM，互不干扰                              │
│                                                                      │
│  ⑥ 实时可中断 Pipeline                                               │
│     StateMachine 支持即时暂停/恢复/终止                               │
│     长时间 LLM 调用和实验执行均可被安全中断                            │
│     应用重启后自动恢复未完成任务                                      │
│                                                                      │
└──────────────────────────────────────────────────────────────────────┘
```

### 7.2 产品竞争力一句话总结

| 维度 | ScholarMind 的回答 |
|------|-------------------|
| **为什么不只是用 ChatGPT？** | ChatGPT 是一个工具；ScholarMind 是一支团队。它不需要你逐个提示，而是自动走完 9 个科研步骤。 |
| **为什么不只是用 AI-Scientist？** | AI-Scientist 是命令行工具；ScholarMind 是完整产品——有前端、有移动端、有对话界面、有多搜索引擎、有 PaperQA2 RAG、有实时推送。 |
| **为什么不只是用 Semantic Scholar？** | Semantic Scholar 只检索论文；ScholarMind 检索论文后还会分析空白、生成 idea、写代码、跑实验、写论文、做评审。 |
| **为什么选择 ScholarMind？** | 因为它是唯一一个覆盖科研全流程的 AI 原生平台——从一句话到一篇论文，全程可追溯、可中断、可降级、可扩展。 |
