# ScholarMind 桌面端前后端对接计划书

## 1. 文档目标

本文档用于指导 `Project\ScholarMind\react-client` 与 ScholarMind 桌面端后端完成正式对接，目标是在 **不修改前端既有 UI 设计与页面布局**、**不修改后端核心流水线逻辑** 的前提下，实现桌面端前后端的稳定联调、可演示运行和后续可持续迭代。

本文档关注的是：

- 前端如何接入现有 FastAPI 后端
- `react-client` 的页面状态如何映射后端任务与产物
- 哪些能力可直接复用，哪些需要增加轻量适配层
- 如何分阶段实施，如何控制风险，如何定义验收标准

本文档不包含：

- UI 重设计
- 页面布局改版
- 9 模块科研流水线的算法改造
- 后端模块执行逻辑重写
- 登录鉴权体系重构

---

## 2. 项目边界与硬约束

### 2.1 前端约束

- 保留 `react-client` 现有路由结构、页面层级、视觉风格、组件组织方式
- 不修改页面布局、不重排主要区块、不替换现有设计语言
- 如需改动，仅允许进行“数据接入型改动”
  - 新增 API 服务层
  - 新增 WebSocket 服务层
  - 新增数据适配层
  - 替换页面中的 mock 数据来源
  - 保持现有交互位置不变

### 2.2 后端约束

- 保留后端现有 FastAPI 主入口、路由结构、数据库模型、状态机、Tracer、9 模块编排顺序
- 不修改 `PipelineOrchestrator` 的核心执行顺序
- 不修改 M1-M9 的核心业务产出逻辑
- 不调整任务生命周期语义
  - `pending / running / paused / review / completed / failed / aborted`

### 2.3 允许的适配性改动

在不改变核心逻辑前提下，允许做以下轻量改动：

- 新增前端 API/WS 封装文件
- 新增前端 store 适配逻辑
- 新增前端 DTO/Adapter 层
- 新增后端只读型辅助接口
  - 读取模块产物内容
  - 查看代码目录树
  - 读取指定文本/JSON 文件
- 新增开发与构建代理配置
- 新增集成文档与验收脚本

## 3. 当前系统现状分析

## 3.1 后端现状

当前后端已经具备完整的桌面端任务中心能力：

- 基于 FastAPI 提供 REST API 与 WebSocket
- 任务创建后立即启动 9 模块科研流水线
- 支持任务状态查询、暂停、恢复、终止、删除
- 支持日志查询、产物查询、评审结果查询
- 支持全局和单任务 WebSocket 推送
- 产物以文件形式落盘到 `backend/workspace/<task_id>/`

后端核心接口已具备：

- `POST /api/tasks`
- `GET /api/tasks`
- `GET /api/tasks/{task_id}`
- `POST /api/tasks/{task_id}/pause`
- `POST /api/tasks/{task_id}/resume`
- `POST /api/tasks/{task_id}/abort`
- `DELETE /api/tasks/{task_id}`
- `GET /api/tasks/{task_id}/logs`
- `GET /api/tasks/{task_id}/output`
- `GET /api/tasks/{task_id}/review-result`
- `GET /api/files/{task_id}/{file_path:path}`
- `WS /api/ws`
- `WS /api/ws/{task_id}`

后端核心优势：

- 任务模型清晰
- 流程状态完整
- 产物有固定落盘目录
- 日志与结果都可追踪

后端当前局限：

- 返回的是“任务流视角”，不是“工作台视角”
- `react-client` 所需的一些页面级数据结构没有直接提供
- 代码仓库仅暴露目录 URL，没有文件树/文件内容接口
- 论文写作页与验证页需要更细的内容拆分，目前仅有聚合结果或文件
- WebSocket 消息与旧前端类型定义存在轻微契约差异，需要在新接入时统一

## 3.2 `react-client` 现状

`react-client` 当前更像一个“科研工作台”原型，而不是一个纯任务详情页。

其特点是：

- 路由颗粒度更细
- 页面分成 12 个工作阶段
- 页面视觉与布局已完成
- Zustand store 中大量使用本地 mock 数据
- 尚未真正接入后端 REST/WS

当前前端主要页面包括：

- 工作台 `workspace`
- 流程总览 `workflow`
- 历史记录 `history`
- 设置 `settings`
- 领域探索 `exploration`
- 文献采集 `literature`
- 信息提取 `extraction`
- 趋势分析 `trends`
- 研究缺口 `gaps`
- 想法生成 `ideas`
- 资料库 `repository`
- 实验设计 `experiment`
- 智能体运行 `agent-run`
- 结果分析 `results`
- 论文写作 `writing`
- 结论验证 `validation`

当前前端主要问题：

- 没有 API 服务层
- 没有 WebSocket 服务层
- store 结构与后端任务结构不是一一对应
- 会话、消息、论文、实验、写作、验证等数据均为静态 mock
- 不能直接消费后端现有 REST 返回

## 3.3 旧 `frontend` 的参考价值

旧 `frontend` 已经实现了基本的 REST + WS 接入，价值在于：

- 可复用其 API 调用模式
- 可复用其 WebSocket 连接策略
- 可复用其任务轮询与局部刷新经验

但旧 `frontend` 不适合作为直接替换方案，原因是：

- 页面模型与 `react-client` 完全不同
- 功能维度更少
- 不符合当前要保留的桌面端设计

结论：

- 旧 `frontend` 适合作为“接口调用参考”
- `react-client` 应作为“正式桌面端 UI”
- 需要在 `react-client` 中建立新的适配层，而不是强行迁移旧 UI

## 4. 总体对接策略

总体策略采用：

**后端任务流不变，前端新增适配层，将任务流映射为工作台状态。**

即：

1. 后端继续按现有 9 模块执行
2. 前端不改变页面结构与视觉设计
3. 在 `react-client` 中新增统一数据接入层
4. 通过 Adapter 将后端任务、日志、产物、评审结果转换为前端页面状态
5. 仅在必要时补充只读型后端辅助接口，不触碰核心业务逻辑

核心思想：

- 后端提供“事实数据”
- 前端负责“视图解释”
- 不在页面层直接拼装后端原始字段
- 所有映射集中到 Adapter 层

## 5. 前后端数据模型映射设计

## 5.1 核心映射原则

- 后端的 `Task` 是单一事实源
- 前端的 `session` 概念映射为后端 `task`
- 前端的 12 阶段是后端 9 模块的视图化拆分
- 页面展示使用“派生状态”，不反向影响后端流水线
- 写作、验证、资料库等页面优先基于后端落盘文件构建只读视图

## 5.2 会话模型映射

前端：

- `sessions`
- `currentSessionId`
- `history`

后端来源：

- `GET /api/tasks`
- `GET /api/tasks/{task_id}`

映射规则：

- `session.id = task.id`
- `session.title = task.title`
- `session.domain = task.description || task.topic`
- `session.updatedAt = task.updated_at`
- `session.stageLabel = current_module + 当前阶段中文标签`

实施要求：

- 历史页直接展示任务列表
- 当前选中的 session 实际上就是当前 task

## 5.3 工作流阶段映射

`react-client` 12 阶段与后端 9 模块不存在严格同构关系，因此采用“视图映射”：

| 前端阶段 | 后端模块/数据来源 | 说明 |
|---|---|---|
| exploration | `task.topic`、`task.description`、M1 前期结果 | 作为研究上下文入口，不单独驱动后端模块 |
| literature | M1 | 直接映射 |
| extraction | M1 产物拆解 | 由文献源数据和摘要派生 |
| trends | M1/M2 派生 | 由文献与 gap 分析结果归纳 |
| gaps | M2 | 直接映射 |
| ideas | M3 | 直接映射 |
| repository | M4 | 直接映射 |
| experiment | M5 | 直接映射 |
| agent-run | M6 | 直接映射 |
| results | M7 | 直接映射 |
| writing | M8 | 直接映射 |
| validation | M9 | 直接映射 |

阶段状态映射规则：

- `not-started` 对应模块 `waiting`
- `in-progress` 对应模块 `running`
- `completed` 对应模块 `completed`
- `risk` 对应模块 `failed`，或评审拒绝，或关键文件缺失

补充规则：

- `exploration/extraction/trends` 不作为独立后端执行节点，仅作为工作台视图页
- 其状态由 M1-M2 的完成度和产物可读性推导

## 5.4 聊天工作台映射

当前 `workspace` 页是对话式主控台，但后端没有真正的聊天会话 API。

规划策略：

- 第一阶段不构造伪聊天后端
- 用户提交研究主题时，调用 `POST /api/tasks` 创建任务
- 创建成功后，将系统反馈渲染为工作台消息
- 后续消息区展示：
  - 任务创建成功提示
  - 当前模块状态摘要
  - 可跳转阶段快捷入口
  - 最近关键日志摘要

约束说明：

- 不新增对话大模型能力
- 不把“聊天输入框”改成别的布局
- 只改变其数据来源与提交动作

## 5.5 日志与实时状态映射

前端来源：

- `runLogs`
- `runProgress`
- `runStatus`
- `workflow stages`

后端来源：

- `GET /api/tasks/{task_id}`
- `GET /api/tasks/{task_id}/logs`
- `WS /api/ws/{task_id}`

映射规则：

- 任务总体状态来自 `task.status`
- 当前模块与模块百分比来自 `task.current_module` 和 `task.modules`
- 实时日志以 WebSocket 增量 + REST 首屏拉取结合方式实现
- 页面初次进入时先拉 REST
- WebSocket 只做增量更新和状态刷新

## 5.6 产物映射

后端已有固定文件产物：

- M1: `m1_literature_review.md`、`m1_sources.json`
- M2: `m2_gap_analysis.json`
- M3: `m3_scored_ideas.json`
- M4: 代码目录 `project*`
- M5: `m5_experiment_plan.json`
- M6: `m6_experiment_results.json`
- M7: `m7_analysis.json`、`m7_analysis_report.md`
- M8: `paper/paper.tex`、`paper/references.bib`、`paper/paper.pdf`
- M9: `m9_review_report.json`、`m9_review_report.md`

当前后端已有的产物接口：

- `GET /api/tasks/{task_id}/output`
- `GET /api/files/{task_id}/{file_path:path}`

计划中的产物读取策略：

- 优先复用 `output` 接口拿 PDF、代码目录、图像 URL
- 文本型/JSON 型模块产物通过新增只读辅助接口读取
- 不在前端直接猜测服务器文件路径

## 6. 推荐新增的适配层结构

在 `react-client` 中新增以下结构：

```text
src/
  services/
    api.ts
    websocket.ts
  adapters/
    taskAdapter.ts
    stageAdapter.ts
    artifactAdapter.ts
    logAdapter.ts
  types/
    backend.ts
```

### 6.1 `services/api.ts`

职责：

- 封装所有 REST 调用
- 统一 `BASE_URL`
- 统一错误处理
- 统一超时与 JSON 解析

建议接口：

- `createTask`
- `getTasks`
- `getTask`
- `pauseTask`
- `resumeTask`
- `abortTask`
- `deleteTask`
- `getTaskLogs`
- `getTaskOutput`
- `getReviewResult`
- `getArtifactContent`
- `getRepoTree`
- `getRepoFile`

### 6.2 `services/websocket.ts`

职责：

- 统一单任务 WebSocket 连接
- 统一重连策略
- 统一消息订阅机制
- 与页面解耦

原则：

- 页面不直接处理底层 socket
- store 订阅统一消息

### 6.3 `adapters/taskAdapter.ts`

职责：

- 将后端 `TaskResponse` 转为前端 `session/stage/current status`
- 把后端状态词汇映射为工作台词汇

### 6.4 `adapters/artifactAdapter.ts`

职责：

- 将后端 JSON/MD/TXT 文件内容转换为页面所需结构
- 文献、gap、ideas、results、writing、validation 均走统一入口

### 6.5 `types/backend.ts`

职责：

- 单独定义后端 DTO
- 避免前端工作台类型与后端接口类型混写

## 7. 后端建议新增的轻量接口

为减少前端猜路径和减少页面耦合，建议新增以下只读型接口。

注意：

- 这些接口不改核心执行逻辑
- 不改变数据库模型
- 不改变模块行为
- 只是在现有产物之上增加只读访问能力

## 7.1 产物清单接口

建议接口：

`GET /api/tasks/{task_id}/artifacts`

职责：

- 返回任务已生成的全部模块产物清单
- 包括类型、模块、文件名、可读 URL、内容类型

用途：

- 前端统一判断页面是否可展示真实内容
- 前端避免手写文件名猜测逻辑

## 7.2 文本/JSON 产物读取接口

建议接口：

`GET /api/tasks/{task_id}/artifact-content?path=...`

职责：

- 读取工作区内特定文本型产物
- 支持 `.json`、`.md`、`.txt`、`.tex`、`.bib`

用途：

- 文献页读取 `m1_sources.json`
- 缺口页读取 `m2_gap_analysis.json`
- 想法页读取 `m3_scored_ideas.json`
- 实验设计页读取 `m5_experiment_plan.json`
- 结果页读取 `m7_analysis.json`
- 写作页读取 `paper.tex`
- 验证页读取 `m9_review_report.json`

## 7.3 代码目录树接口

建议接口：

`GET /api/tasks/{task_id}/repo/tree`

职责：

- 返回代码目录树
- 用于 `repository` 页左侧文件树

## 7.4 代码文件内容接口

建议接口：

`GET /api/tasks/{task_id}/repo/file?path=...`

职责：

- 读取指定代码文件内容
- 用于 `repository` 页右侧预览

## 7.5 评审完整结果接口

建议接口：

`GET /api/tasks/{task_id}/review-report`

职责：

- 返回完整的 `m9_review_report.json`
- 不仅是聚合评分，还包括 reviewer 维度与缺失引用信息

用途：

- 支撑 `validation` 页的结论核查与风险提示

## 8. 页面级对接方案

## 8.1 LoginPage

目标：

- 暂不接入正式鉴权
- 保留当前 UI 和输入框布局

方案：

- 继续使用本地登录态
- 登录后跳转工作台
- 后续如需接入鉴权，单独立项，不纳入本次对接主线

## 8.2 MainChatWorkspacePage

目标：

- 保留聊天式工作台外观
- 将提交动作改为创建任务

方案：

- 输入研究主题后调用 `POST /api/tasks`
- 成功后：
  - 追加用户消息
  - 追加系统消息
  - 设置当前 session/task
  - 根据任务状态生成快捷跳转按钮
- 不引入新的对话后端协议

验收标准：

- 输入研究主题后可真实创建任务
- 工作台消息区显示真实任务初始化反馈

## 8.3 WorkflowOverviewPage

目标：

- 使用真实任务阶段状态替换 mock

方案：

- 从当前 task 的 `modules` 派生 `stages`
- 保留页面排版
- 点击阶段仍按现有路由跳转

## 8.4 HistoryPage

目标：

- 以真实任务列表替换 mock 会话列表

方案：

- 页面加载时调用 `GET /api/tasks`
- 选中一条记录时设置 `currentSessionId = task.id`
- 搜索框仍在前端本地过滤

## 8.5 LiteraturePage

目标：

- 展示真实文献采集结果

方案：

- 进度条来自 M1 状态
- 文献表来自 `m1_sources.json`
- 文献综述摘要来自 `m1_literature_review.md`
- 保留现有筛选控件
- 筛选仅做前端过滤，不反向驱动后端

说明：

- “开始采集”按钮可采用以下策略之一
  - 若当前无任务，则要求先在工作台创建任务
  - 若当前有任务，则跳转/提示该任务已进入真实流程

## 8.6 InformationExtractionPage

目标：

- 在不改 UI 的情况下，让该页有真实内容来源

方案：

- 由 M1 文献结果与摘要派生提取结果
- 第一阶段可先以只读聚合视图展示
- 第二阶段若需要更细粒度结构，可增加轻量整理函数，不改后端模块

## 8.7 TrendAnalysisPage

目标：

- 将趋势页改为“基于 M1/M2 结果的派生分析页”

方案：

- 使用 M1 文献年份、来源、关键词
- 使用 M2 gap 输出中的主题归纳
- 前端生成趋势事件与热点方向

说明：

- 本页主要由前端派生构建，不要求后端新增独立趋势模块

## 8.8 ResearchGapsPage

目标：

- 展示 M2 gap 分析结果

方案：

- 读取 `m2_gap_analysis.json`
- 通过 adapter 映射为当前页面所需 `ResearchGap[]`
- 保留现有卡片布局

## 8.9 IdeaGenerationPage

目标：

- 展示 M3 生成的候选 idea

方案：

- 读取 `m3_scored_ideas.json`
- 映射为当前页面的 `IdeaCandidate[]`
- 保留推荐标记、评分维度与选择逻辑

## 8.10 RepositoryPage

目标：

- 将 mock 文件树替换为真实代码仓库内容

方案：

- 左侧文件树来自 `GET /repo/tree`
- 右侧内容来自 `GET /repo/file?path=...`
- 若接口未就绪，则先展示代码目录下载/打开入口

关键约束：

- 不修改当前左右分栏布局
- 不改文件树交互结构

## 8.11 ExperimentDesignPage

目标：

- 展示真实实验计划

方案：

- 读取 `m5_experiment_plan.json`
- 映射为页面当前 experiment design 结构
- 保留表单布局，但初期建议只读展示

说明：

- 当前后端并没有“保存实验设计并重新驱动任务”的独立接口
- 因此此页首期以“展示真实计划”为主，不承担重新编排任务的职责

## 8.12 AgentRunPage

目标：

- 成为桌面端实时运行监控页

方案：

- 初次进入时加载 `GET /tasks/{id}` 和 `GET /logs`
- 按任务订阅 `WS /api/ws/{task_id}`
- 使用任务模块进度映射运行步骤
- 使用日志流更新右侧日志区

验收标准：

- 能看到真实任务进度
- 能看到真实日志刷新
- WebSocket 断开后可自动重连

## 8.13 ResultsAnalysisPage

目标：

- 展示 M7 结果分析与评估结论

方案：

- 读取 `m7_analysis.json`
- 转为指标对比、解释文本、误差案例

## 9. WebSocket 对接方案

## 9.1 接入原则

- 每次只对当前选中的任务建立单任务订阅
- 保留自动重连
- WebSocket 仅负责增量更新
- 首屏状态仍以 REST 为准

## 9.2 消息处理策略

统一处理消息类型：

- `progress`
- `result`
- `need_review`
- `error`
- `completed`

处理原则：

- `progress`
  - 刷新当前任务阶段状态
  - 追加日志摘要
- `result`
  - 标记当前模块已完成
  - 触发相关页面数据失效刷新
- `need_review`
  - 当前版本先展示提示，不强制进入人工审核流程
- `error`
  - 标记任务或模块风险状态
  - 在运行页和流程页显示异常
- `completed`
  - 触发结果页、论文页、验证页可用状态

## 9.3 注意事项

- 不直接依赖 WS 里的 percent 字段作为唯一进度来源
- 关键状态以 `GET /tasks/{id}` 为准
- 前端要兼容断连和重复消息

## 12. 风险分析与控制策略

## 12.1 风险一：前端页面模型比后端任务模型更细

表现：

- 12 页面并不等于 9 模块

影响：

- 页面会出现“无直接接口数据源”的情况

控制策略：

- 建立统一 Adapter
- 对 `exploration/extraction/trends` 采用派生视图策略
- 明确这些页面不是独立后端模块

## 12.2 风险二：页面需要的粒度比后端现有接口更细

表现：

- 资料库页需要文件树
- 写作页需要章节拆分
- 验证页需要完整 reviewer 信息

影响：

- 仅靠现有 `/output` 不足以支撑

控制策略：

- 增加只读辅助接口
- 不动核心逻辑，只开放读取能力

## 12.3 风险三：WebSocket 与 REST 契约不一致

表现：

- 增量消息字段与页面期望不完全等同

影响：

- 运行页实时状态可能错乱

控制策略：

- 统一在前端 websocket adapter 中标准化消息结构
- 页面层只消费标准化后的消息

## 12.4 风险四：任务切换导致页面状态串扰

表现：

- 从一个 task 切换到另一个 task 时，旧页面数据残留

控制策略：

- 所有页面级数据按 `currentSessionId` 隔离
- 切换任务时清理局部缓存
- 所有异步请求绑定当前 task id

## 12.5 风险五：写作页和验证页超出当前后端天然支持范围

表现：

- 后端产物是文件，不是前端编辑态文档模型

控制策略：

- 首期以只读真实展示为目标
- 不在本次对接中追求在线编辑回写

- 产物类页面依赖任务主链路稳定后再接
- 资料库、写作、验证属于更细粒度读取能力，应放在后期

## 16. 结论

本次对接的正确方向不是改造 `react-client` 的设计，也不是重写后端流水线，而是：

**以最小侵入方式建立“后端任务流 -> 前端工作台”的适配层。**

最终实施原则应始终保持：

- UI 不动
- 布局不动
- 后端核心逻辑不动
- 通过服务层、适配层、只读辅助接口完成接入

只要按本文档的阶段推进，`react-client` 可以在保持现有桌面端设计表达的前提下，逐步替换掉 mock 数据，最终成为 ScholarMind 的正式桌面前端。
